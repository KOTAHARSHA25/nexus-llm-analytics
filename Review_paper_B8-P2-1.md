*Abstract—The use of data analysis plays a vital role in the decision-making process, and for many users, conducting complex analyses often poses barriers due to their lack of programming or statistical expertise. Traditional tools typically require manual coding and domain-related knowledge, which can limit access and generate inefficiencies across all industries. Recent progress has been made with the introduction of Large Language Model (LLM) capabilities, facilitating greater automation of the data analysis workflow and providing users with additional means of interacting with data analysis in a user-friendly manner. Unfortunately, most current systems continue to limit advanced data analysis capabilities by providing little in the way of autonomy, flexibility and an adequate error correction methodology during the execution phase of the analysis.This paper introduces the Domain-Flexible Autonomous Data Analysis Agent (DF-Agent) to address the limitations of most existing systems. DF-Agent automatically generates executable Python or SQL code based on a plain-language query, executes the code within a sandboxed environment, verifies the outcome of the code execution and iteratively corrects any errors detected through automated fault tolerance mechanisms. The DF-Agent architecture consists of several modules, including the Planner, which breaks the task into smaller components; Code Synthesizer for generating programmatic code, Executor for executing the programmatic code with controlled execution; and Verifier for confirming the results generated by the comprehensive data analysis process. The combined functionality of intelligent code generation, runtime feedback, and iterative verification loops allows the DF-Agent to support domain-independent, consistent, reliable, repeatable analytics. As a result, non-technical users can accurately conduct clear, transparent analyses using the DF-Agent, freeing up the user from acting as the manual input of data into the analysis process.*

*Keywords—Autonomous Agent, Data Analysis, Large Language Models, Program Synthesis, Reproducibility, Natural Language Interface*

I. INTRODUCTION

Digital data is growing exponentially across industries. As a result of this surge in volume, there has been a concerted effort to create tools and systems that can analyse this data quickly and accurately, regardless of industry. Businesses use data analysis, as an example, as a way to gather competitive intelligence. Researchers use it to conduct scientific research. Health care and education areas also depend on data analysis. Decision-makers increasingly use these tools to extract useful information from vast, complex datasets. Traditionally, conducting the necessary analyses requires some level of proficiency in programming and statistics. In addition, there is a requirement to have knowledge of the specific tools associated with a given industry segment. As such, many individuals who do not possess a technical background or relevant experience may be excluded from using these tools and the analysis results. The system referred to in this article is a domain flexible autonomous data analysis system, or DF agent. Using DF agents, users are able to input data or queries into an application using plain English and the system will convert them into executable Python or SQL code, allowing for complete data analysis without requiring technical knowledge.The program executes code using a secured environment, verifies results and subsequently outlines each result and displays all results visually. By combining vast automated processes (some of which include an extensive array of built-in Fixes & Checks) the whole methodology provides enhanced ease-of-use and reliability for all analytics generated by the Program. This makes it much easier for people to replicate the results. DF Agent provides a very unique and flexible solution that can grow with a user's need and adapt to an ever-changing environment. It also provides support for Data Driven Decisions across many industries.

# III. Problem Formulation

The goal of the Autonomous Data Analysis Agent is to deliver a completely self-sufficient, intelligent, and thorough way of analyzing data through an interaction using natural language; it does this by accepting natural language queries and raw dataset input from users and employing large language models in conjunction with a structured multi-agent collaboration framework for the complete synthesis, validation, and execution of analysis coding, creation of informative visualizations, and creation of structured written reports. The goal of this proposed system is to enable full end-to-end analytical capabilities using only natural language input and therefore users will not have to be proficient in programming or possess a high level of statistical capability. To achieve this goal, the system’s architecture will coordinate numerous specialists reasoning agents such as: Data Analyst Agent (the agent that performs data analyses), Code Validation Agent (the agent that validates the code generated from data analysis), Visualization Expert Agent (the agent that generates data visualizations), and Technical Report Generation Agent (the agent that writes the technical reports documenting the data analysis results), using an Adaptive Multi-Agent Orchestration Framework (A-MAOF) which provides a robust means of ensuring modularity, execution reliability, and reasoning efficiency. In addition, the system supports multi-modal intelligence through enabling reasoning, verification, and summation to be carried out across multiple Large Language Models (LLMs) such as Llama 3.1 8B, Phi-3-mini, and TinyLlama. Generated Python and SQL computational code is executed within a secure sandboxed execution environment, which ensures that analytical operations are executed safely, in a controlled manner, and are fully reproducible.

IV. Methodology

The Domain Flexible autonomous LLM Analytics framework differs from traditional analytic systems through the use of adaptive orchestration, modular agent specialization and iterative verification as well as integrated search and identify, process and review techniques. Conventional analytics based on LLM products rely heavily on fixed reasoning models or retrievable documents from established retrieval pipelines, whereas Nexus will dynamically choose from various methodologies including which Language Model(s) to utilise, the method(s) of execution and the level of verification performed based upon the Query Complexity Level(s) and Analytical Intent.The intelligent-routing system utilizes a rule-based router to examine user queries prior to executing them. The rule-based router classifies a query's complexity level (low or high) based on predefined rules about how complex a user query could be; determined by factors such as query length, the number of analytical operations requested, the presence of aggregations or joins, and the requirement for multi-step reasoning When a low-complexity user request arrives at the router, the output will typically always be a response generated using lightweight models; while any format generated via a high-complexity user request will require more time and resources from the router using either a heavier model or by tapping into one or more databases simultaneously. The resource consumption for low-complexity vs. high-complexity user requests is optimized via the intelligent-routing mechanism providing access to both lightweight user requests as well as higher-quality responses produced by larger models after performing aggregations from multiple haptic data sources and executing the analytical models required during the information transformation process thus maximising usability and output quality.Additionally, Nexus includes an architecture based upon multiple types of agents that are designed for analytical purposes utilizing specialised methodologies including statistics, machine learning, time-series analysis, and document-centric tasks creating a more holistic approach to the data analysis process.Code created using tools developed by this organisation creates secure and reproducible results by running code execution in isolated environments, with limited access to all resources and no outside network connectivity. With the ability to execute sandboxed code and perform selective review loops, the generated Analytical Workflows will not only be safe but also correct thereby minimising the concerns about reliability associated with executing LLM outputs.Moreover, the inclusion of Semantic Query Enhancement, Retrieval-Augmented Generation and Citation Tracking will substantially improve the contextual accuracy and transparency of the use of previously unencountered Data Sets.

Figure1 illustrates the autonomous multi-agent execution pipeline of the Nexus LLM Analytics framework. Users submit analytical requests by entering queries in natural language that detail the analyses and visual outputs they would like to receive. Once the Multi-Agent Orchestration Layer receives this request, it interprets it and decomposes the user's intent into a series of actionable tasks that can then be completed by the appropriate specialist agents based on multiple factors including complexity.The execution pipeline consists of 4 core specialist agents working in concert. The Data Analyst Agent reviews and prepares datasets for analysis; performs statistical analysis; and creates executable code. The Retrieval-Augmented Generation (RAG) Agent provides context and historical references necessary to complete the analysis, as well as other reference material that can be used to support a more comprehensive approach to analytical reasoning. The Code Reviewer or Validator Agent ensures that the code created by the Data Analyst Agent is correct, safe, and resource compliant. Finally, the Visualization Agent creates charts and figures for the user that are aligned with the analytical goals established by the user.All coordination among these agents is performed by the local LLM-based inference engine, which selects the appropriate language models to use on a dynamic basis when executing reasoning and validation tasks. All execution takes place within a controlled, isolated, sandboxed environment, which provides security, privacy, and reproducibility. The shared vector memory allows continual access to previously completed analyses, thereby facilitating incremental learning and reproducible analytical workflows.

The Nexus LLM Analytics framework has limitations as well as strong adaptability, modularity, and dependability. One limitation is that the complexity-based orchestration approach utilizes heuristic scoring based on query properties. While transparent and efficient, there can be many different optimal threshold values for various domain/workload combinations, so therefore, additional research may be done using calibration of threshold values based on data or learning-based routing of models.A second limitation is that the present evaluations are concentrating on evaluating the vector-based retrieval (VBR) technique. Hybrid retrieval and re-ranking techniques have also been established but will require additional testing for systematic evaluation.

# V.System architecture and implementation

The Domain-Flexible Autonomous Data Analysis Agent (DFA-DAA) is a large-language model-driven multi-agent collaboration project to automate the analysis of complicated data sets using conversational language queries. DFA-DAA makes use of multiple Large Language Models (LLMs), a Framework for Agentic Orchestration, Secure Sandboxed Execution and other approaches to allow for real-time and explainable analytics (i.e., non-domain dependent). With this solution, anyone can analyze data without having to code or have statistical experience to find insights within various datasets.Secure Sandbox Execution is an effective way to protect the integrity and the security of your Analytical Data, as it allows you to isolate the execution of your analyses from the storage and processing of the Host Operating System.Use of this Framework will ensure that no harmful activities occur on the Host Operating Systems, and that full Compliance with strict Data Protection Laws is maintained. In addition, as a functioning framework for Agentic orchestrations, Reasoning Agents can dynamically choose the appropriate models for their analyses, appropriately allocate work between Reasoning Agents, and provide full visibility of how the outputs and processes are traceable to clients so that Users are fully aware of what is being analysed. It uses Persistent Vector Memory to store the analytical context to facilitate re-use of Knowledge, Incremental Learning, and Continuity between user sessions.

Figure2. End-to-End Data Flow and Processing Pipeline in the Multi-Agent LLMSystem.

Figure2 describes the end-to-end workflow and processing methodology of the Proposed Autonomous Multi-agent (LLM) Based Analytical System. The workflow and data processing methodology of the Autonomous Multi-Agent (LLM) Based Analytical System consist of Four Logical Levels: Input; Data Preparation; Autonomous Multi-Agent Data Analytic; and Output. The data processing methodology of the Autonomous Multi-Agent (LLM) Based Analytical System begins at the Input Level where Users can send Natural Language Chat (NLCs) or upload Structured/Unstructured Data Files (CSV, Excel, Text and PDF) for analysis. Regardless of which method the user chooses to submit their analysis request, all requests triggered by the user's action are processed through an Application Programming (AP) Gateway that performs User Authentication and authorizes secure messages between the user and the LLM-Based Analytic System. In addition, this approach allows Users to express Complex Analysis Requests in Natural Language Form (NLF) with no need for programming skills or domain knowledge on the part of the user, thus enabling machine-readable formats for human input.Once collected, the data entered into the Data Pre-Processing Layer, which cleans and validates the data. This step focuses upon the schema and type inference and header extraction, as well as identifying which column each piece of the data belongs to. Data that does not meet the necessary requirements and that does not meet standard formatting will be stripped out as part of this process. Cleaned and validated data will be stored in separate store places; all other forms of unstructured text data will be represented as vector embeddings and subsequently stored in a vector memory database. The use of vector memory allows for semantically indexing, recalling, and reusing previous findability so retrieve as needed, which prevents repetitiveness in the retrieval of computational results. As such, thus layer ensures the dataset is standardized, trustworthy, and available for analysis without any redundancies. The intelligence component of this data processing phase consists of the Autonomous Multi-Agent Analytics and code generation agents that work collaboratively within a secure sandboxed space powered by LLMs. The code generation agent receives the analytics work products and translates them into code that can then be executed for data processing, statistical analysis, feature extraction or predictive modeling. Also included within the layer, is an agent for choosing and utilizing visualization types such as histograms, scatter plots, correlation heat maps, and dashboards, all based upon the type of data inputted.The system executes all code within a sandbox environment that limits the system's ability to make new calls to external services and restricts access to only those (‘whitelisted’) library files. After completing the analysis, the output layer converts the output into a format that is meaningful to the user. Additionally, the system creates Graphical output, Tables, Calculated Values and Automatic Technical Reports at the conclusion of the analytical process, providing the Reasons/Logs used in constructing the Technical Reports and showing Graphical Evidence created, along with the Interpretation of Results.The above-mentioned Figure 1 clearly illustrates a new level of technical innovation established by the proposed architectural framework in the successful demonstration of the Privacy-encouraging and Autonomous Analytic Workflow.

Figure3.Multi-Agent Execution Pipeline

Figure3 illustrate the autonomous multi-agent execution pipeline of the Nexus LLM Analytics framework, consists of several sequential steps that begin with the query submitted by users in the form of natural language queries articulating their desired analyses and visual outputs. Once the query is received, the Multi-Agent Orchestration Layer interprets it into individual, actionable tasks that can be assigned based on the type of analysis required (as defined by the user) to specific types of specialist agents and will monitor and manage how these tasks are executed in relation to each other.

There are four distinct specialist agents that work together through an automated multi-agent execution pipeline. The Data Analyst Agent evaluates and preprocesses datasets, performs statistical analyses, and generates executable codes; the Retrieval-Augmented Generation (RAG) Agent collects contextual knowledge, historical references, and documentation to support improved analysis of the data; the Code Reviewer/Validator Agent reviews and checks the correctness and safety of the generated code, as well as ensuring that it meets defined resource requirements; and, lastly, the Visualization Agent selects and generates charts and other graphical representations of the data and results that best meet the objectives of the analysis.

The relationships between these specialist agents and how they interact with each other are managed through a local inference engine (LLM) that contains all the major components of the Nexus LLM Analytics framework. The inference engine will select the best language model for any particular reasoning or validation task in an isolated sandboxed environment that offers the highest level of security, privacy, and reproducibility. The agents sharing a common vector memory will enable the agents to continue working together from one analysis session to the next, as well as allowing for continuous learning from incremental contributions throughout the entire analysis process.

# VII. Conclusion

The Domain-Flexible Autonomous Data Analysis Agent utilizes large language models and agent-like automated intelligence will change the way AI is used in data analysis. Using multi-agent collaboration, retrieval-augmented generation, and the ability to generate dynamic executable code from Natural Language User Requests (NLURs), this system creates executable analytical tasks (technically called “workflows”) from previously entered NLURs. This allows those without technical knowledge to perform complex, non-domain specific analyses easily, through the use of these system-created workflows, while providing for automated, explainable, and adaptive accuracies due to the intelligent orchestration of the work generated by multiple subject area experts. In turn, this project creates an expanded scalable potential for autonomous data-driven decision-making and the intelligent assistance of analysis through a new architecture.
